# 기차 라우팅·예약 엔진 (C)

> 그래프 최단경로 탐색 + 다구간 출발시각 스케줄링 + Red-Black Tree 예약 관리를 하나로 결합한 라우팅·예약 처리 엔진

---

## 문제

기차 예매 한 건을 처리하려면 세 가지 문제를 한 번에 풀어야 했다.
1. 출발–도착 도시 사이의 **경로**를 찾고,
2. 그 경로를 구성하는 각 **구간(leg)마다 가장 빠른 출발편**을 시간 순서대로 연결하고,
3. 확정된 예약을 **빠르게 조회·관리**할 수 있게 저장.

단순 최단경로 탐색만으로는 부족했다. 경로를 찾아도 "언제 타고 언제 갈아타는가"라는 시간축 스케줄링이 빠지면 실제 예매가 성립하지 않는다.

## 해결

- **경로탐색** — 26개 도시 인접리스트 그래프에서 다익스트라로 최단경로 산출. 간선의 실제 기하 거리를 가중치로 사용하는 **가중치 기반 최단경로**.
- **스케줄링** — `일 × 도시 × 경로 × 시각`의 **4차원 출발시각 테이블**에서, 각 구간마다 "이전 구간 도착 시각 이후"의 가장 빠른 출발편을 탐색(`findFastestDeparture`). 한 구간의 도착 시각을 다음 구간의 기대 출발 시각으로 전파하여 **다구간 환승 일정을 시간 순서대로 연결**.
- **예약 관리** — 확정된 각 구간 예약을 **Red-Black Tree에 직접 구현하여 저장**, O(log n) 삽입·조회 보장.
- **파이프라인** — 예매 요청 파싱 → 최단경로 → 구간별 빠른 출발편 → RB트리 예약까지 한 흐름으로 처리.

## 아키텍처

```
[예매 요청 파싱]  name, src, dst, time-date
        │
        ▼
[경로탐색]  dijkstra() → findShortestPath()     travelPath.cc
        │   가중치(거리) 기반 최단경로, 경로 = 도시 시퀀스
        ▼
[스케줄링]  findFastestDeparture() (구간별 반복)  server.cc / departTime.cc
        │   4D 출발시각 테이블에서 leg별 최速 출발편
        │   도착 시각 → 다음 leg 기대 시각으로 전파
        ▼
[예약 관리]  makeReservation() → rbInsert()       reserve.cc
            Red-Black Tree (직접 구현)
```

- `travelPath.cc` — 그래프 생성, 다익스트라, 최단경로 복원
- `departTime.cc` — 4차원 출발시각 테이블 생성
- `server.cc` — 요청 파싱 + 경로·스케줄링·예약 오케스트레이션 (엔트리포인트)
- `reserve.cc` — Red-Black Tree 삽입/삭제/탐색/출력 직접 구현

## 규모 / 수치

- 도시 노드 **26개**, 노드당 단방향 경로 최대 **10개** (양방향 금지)
- 출발시각 테이블 **31일 × 26도시 × 10경로 × 5시각 = 40,300 엔트리** (4차원 배열)
- 다익스트라 O(V²), 예약 저장 O(log n) (RB트리)
- 다구간 환승 시 구간별 시간 전파로 일·시각이 누적 진행

## 기술 키워드

`C` · `그래프/인접리스트` · `다익스트라(가중치 최단경로)` · `다구간 스케줄링/시간 전파` · `Red-Black Tree 직접 구현` · `4차원 테이블 탐색`

---

### AMHS 직무 연결 포인트
AMHS S/W 핵심 직무(반송 로직 · 스케줄링 · 경로탐색)와 거의 1:1 대응.
- **경로탐색** ↔ 가중치 다익스트라
- **스케줄링** ↔ 구간별 출발시각 전파 기반 다구간 일정 연결
- **자료구조 깊이** ↔ Red-Black Tree 직접 구현

> 표기 주의: 소켓·스레드·동시성이 없으므로 "동시성 예약 서버"가 아니라 **"라우팅·예약 엔진(처리 파이프라인)"**으로 표기.
